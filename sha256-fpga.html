Overview

SHA-256 is a cryptographic hash algorithm that maps an arbitrary-length message to a unique 256-bit digest. The design satisfies all three security properties — preimage resistance, second-preimage resistance, and collision resistance — making it suitable for authentication, file integrity verification, and blockchain protocols.

This project implements both the core SHA-256 algorithm and Bitcoin's double-SHA256 mining scheme in SystemVerilog, targeting an Arria II GX FPGA. The Bitcoin accelerator parallelizes 16 nonce candidates simultaneously, reusing the SHA-256 core across three pipelined phases, with all results verified correct against reference vectors in ModelSim.

  • IDLE — Registers and working variables (A–H) are initialized to the SHA-256 initial hash constants. A done signal is raised upon completion and the FSM waits here for a start pulse.
  • READ — The message is read word-by-word from memory, incrementing the address pointer each clock cycle. Words are stored into a message block array. The final block is zero-padded with a 0x80 byte separator at the NUMOFWORDS offset and the remaining positions filled with h0 (zero) padding per the SHA-256 spec.
  • BLOCK — Iterates over the number of 512-bit blocks required to cover the full message. The first 16 message words are loaded, then the schedule is expanded to 64 entries (W[16] optimization) before transitioning to COMPUTE.
  • COMPUTE — Executes all 64 SHA-256 compression rounds using the Ch, Maj, Σ0, and Σ1 functions. Working variables A–H are updated each round and the intermediate hash values H[0–7] are accumulated after every block.
  • WRITE — The final H[0–7] digest values are output. The FSM returns to IDLE and asserts done.

Optimization Note — The W[16] rolling-buffer optimization was partially implemented. Correct A–H values were confirmed through the 2nd block; divergence appeared from the 3rd block onward in the general SHA-256 path. The w[16] optimization was successfully applied in the Bitcoin hashing path where fewer consecutive A–H iterations are required.

02

Bitcoin Hashing

Three-Phase Architecture

  • Phase 1 — Initial SHA — Identical to the core SHA-256 flow. The 19-word Bitcoin block header is read from memory and stored. SHA-256 is applied to the first 16 words to produce intermediate hash values H[0–7], which seed the following phases.
  • Phase 2 — Nonce Parallelization — 16 candidate words are constructed in parallel. Each word is composed of the last 3 words from memory, a nonce value ranging from 0 to 15, and standard SHA-256 padding. All 16 SHA computations run concurrently, producing 16 independent sets of hash values.
  • Phase 3 — Final SHA — A third SHA-256 pass is run where the intermediate hash values from Phase 2 serve as the working constants, and the schedule words are derived from those same hash values. This yields the final 16 candidate block hashes, one per nonce.

Simulation Result — All 16 Bitcoin hash outputs (B0[0]–B0[15]) matched reference values. Total clock cycles: 444. Verified in ModelSim with the tb_bitcoin_hash testbench on the 19-word input.
