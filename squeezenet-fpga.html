Overview

This project implements a complete machine learning to hardware deployment pipeline for SqueezeNet — a compact CNN achieving AlexNet-level accuracy with 50× fewer parameters — on the Xilinx Kria KV260 FPGA. The pipeline spans PyTorch model training, quantization-aware training (QAT), C++ HLS optimization, and PYNQ-based deployment.

The accelerator uses 8-bit fixed-point Q3.4 arithmetic (ap_fixed<8,4>), achieving 4× model compression while maintaining 90% classification accuracy on CIFAR-10. Key optimizations include tree reduction for convolution achieving initiation interval II=1, and ping-pong DDR buffering for conflict-free memory management across 15 layers.

The project also navigated two significant engineering challenges: quantization divergence when jointly training weights and activations, and a critical rounding/truncation mismatch between PyTorch training and the default HLS fixed-point behavior — which collapsed hardware accuracy to 20% before being identified and corrected.

Training Pipeline

Three-Stage QAT Approach

FP32 Baseline
Full-Precision Training — 100 Epochs
Standard SqueezeNet trained on CIFAR-10 (upsampled 224×224) with SGD + cosine annealing, data augmentation (random flips, crops, color jitter). Achieves 86.98% test accuracy. Establishes the accuracy ceiling before quantization.
QAT Stage 1
Weight-Only Quantization — 5 Epochs
Apply Q3.4 quantization to model weights only, keeping activations in full precision. Fine-tuned with Adam at lr=1e-4. Stabilizes the weight distribution before activations are introduced, avoiding the divergence seen with joint quantization.
QAT Stage 2
Progressive Activation Quantization — 15 Epochs
Gradually introduces activation quantization via a temperature schedule decreasing from 5.0 (soft) to 0.0 (hard). The straight-through estimator (STE) is used for gradient computation through the non-differentiable quantization operation. Temperature annealing prevents abrupt accuracy drops.
QAT Stage 3
Hard Quantization Fine-Tuning — 10 Epochs
Full hard quantization of both weights and activations with SGD at lr=1e-5. Achieves 81.55% accuracy — versus 23% with naive joint quantization. Final weights and biases are exported as INT8 binary files for FPGA loading.
Challenge — Joint Quantization Divergence

Simultaneously quantizing both weights and activations during QAT caused severe model divergence, with accuracy collapsing to ~23%. The combined quantization noise created optimization instabilities that prevented convergence.

Solution — Progressive Sequential Quantization

Adopting the three-stage approach above — weight-only first, then temperature-annealed activation introduction, then hard fine-tuning — successfully maintained 81.55% accuracy after full Q3.4 quantization.

HLS Implementation

Fixed-Point Type Definitions

// Q3.4 format: 8-bit, 4 integer bits (incl. sign), 4 fractional bits
// Range: [−8, 7.9375], Resolution: 0.0625 (2⁻⁴)
typedef ap_fixed<8, 4, AP_RND_CONV, AP_SAT> data_t;
typedef ap_fixed<8, 4, AP_RND_CONV, AP_SAT> weight_t;
typedef ap_fixed<8, 4, AP_RND_CONV, AP_SAT> bias_t;

// Wide accumulator prevents overflow during MAC operations
typedef ap_fixed<24, 12, AP_RND_CONV, AP_SAT> acc_t;

AP_RND_CONV implements convergent (banker's) rounding, matching PyTorch's torch.round(). AP_SAT matches torch.clamp(). The 24-bit accumulator prevents overflow when summing across input channels.

Convolution Tree Reduction (II=1)

  • Problem — Naive loop-carried accumulation creates inter-iteration dependencies, preventing the HLS compiler from achieving II=1 on the inner channel loop.
  • Solution — Separate partial sum registers for each of the 9 kernel positions (ARRAY_PARTITION complete) allow parallel accumulation across input channels. A DEPENDENCE inter=false pragma informs the compiler no cross-iteration dependency exists.
  • Tree reduction — After the channel loop, the 9 partial sums are reduced in log₂ depth: 4 level-1 pairs → 2 level-2 sums → final bias addition. This eliminates the carry chain delay that would otherwise limit Fmax.
Challenge — Truncation vs. Rounding Mismatch

After initial hardware deployment, the FPGA achieved only 20% accuracy (2/10 images) despite the software model achieving 81.55%. Layer-by-layer comparison using a NumPy reference implementation revealed maximum output deviations of 2.56 (41 LSBs) starting from the very first convolution. Root cause: default ap_fixed uses truncation (round toward −∞), while PyTorch QAT used round-to-nearest. This seemingly minor difference accumulated catastrophically across 15+ layers.

Solution — AP_RND_CONV + AP_SAT Mode

Specifying explicit rounding and saturation modes in every fixed-point type definition (as shown above) made hardware arithmetic exactly match PyTorch's behavior. Hardware accuracy immediately recovered to 90% on the 10-image test set — exceeding the 81.55% software QAT baseline due to favorable rounding on specific samples.

03

Memory & Control

Ping-Pong DDR Buffering

  • Two 1.6 MB DDR buffers (buffer_a and buffer_b) alternate as source and destination across layers, preventing read/write conflicts without requiring on-chip BRAM large enough to hold full feature maps.
  • 740K parameters stored contiguously in DDR in layer order (737,568 weight bytes + 2,986 bias bytes), loaded once at startup via PYNQ DMA.
  • Fire module squeeze buffer — the squeeze layer's output is stored in a dedicated DDR region at offset 1.5 MB within buffer_a to avoid BRAM overflow, then both expand paths (1×1 and 3×3) read from it in parallel.
  • Expand channel interleaving — expand 1×1 writes to output channels 0–E₁−1, expand 3×3 writes to E₁–E₁+E₃−1, and ReLU is applied in-place using burst-buffered DMA transfers.
